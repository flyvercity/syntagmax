# Spec Critique: TOML Format Support for Publish Configuration

## Executive Summary

The specification `docs/specs/publish-toml-support.spec.md` outlines a clear, highly valuable enhancement to support TOML files as an alternative format for publish configuration. Syntagmax already utilizes TOML for global config files (`config.toml`), so extending this support to `publish.toml` creates design consistency and improves developer user experience.

The proposed solution has a clean separation of concerns: format detection/parsing is handled in `load_publish_config()`, file lookup/conflict detection in `resolve_publish_file()`, and orchestration in the configuration fallback resolver.

However, we have identified a few key points under both lenses that must be updated before implementation. Most notably, the helper `resolve_publish_file()` should return the full path (not relative to the queried directory) to avoid complex path reconstruction logic in the fallback resolver, extension case-insensitivity must be handled, and `.is_file()` should be used instead of `.exists()` to prevent issues with directories of the same name.

**Verdict**: ⚠️ **PROCEED WITH UPDATES**

---

## Product Lens Findings

### 1d. Edge Cases & User Experience
*   **P1: Support for `.yml` and Case Insensitivity in Conflict Checks**
    *   *Finding*: The specification states that a per-record `publish` field detects format by extension (`.yaml` / `.yml` vs `.toml`). However, Task 2's conflict helper `resolve_publish_file` only checks for `publish.yaml` and `publish.toml`. If a user has `publish.yml` and `publish.toml` in the same directory, it will bypass conflict detection and fail or behave unexpectedly.
    *   *Suggestion*: Expand the conflict check to look for `publish.yaml`, `publish.yml`, and `publish.toml`. If any YAML/YML variant conflicts with TOML, raise a conflict error.
*   **P2: Case-Insensitive Extension Support**
    *   *Finding*: The suffix comparison (e.g. `resolved_path.suffix`) is case-sensitive by default in Python's `pathlib`. Users using `.YAML` or `.TOML` would trigger the "unknown extension" `FatalError`.
    *   *Suggestion*: Ensure suffix matching is case-insensitive (e.g., using `resolved_path.suffix.lower()`).

---

## Engineering Lens Findings

### 2a. Architecture Soundness
*   **E1: `resolve_publish_file` Path Type and Return Value**
    *   *Finding*: Task 2 specifies that `resolve_publish_file()` should return the path *relative* to the checked directory (e.g., `Path('publish.yaml')`), or `None`. This forces the caller in the fallback chain to reconstruct the path (e.g. prepending `.syntagmax/` or `../.syntagmax/`). This introduces redundant logic, complexity, and risk of bugs in the fallback chain.
    *   *Suggestion*: Have `resolve_publish_file()` return the full resolved path of the found file (e.g. `directory / 'publish.yaml'`), which can then be directly passed to `load_publish_config()` without any path reconstruction.

### 2b. Failure Mode Analysis
*   **E2: Directory / Non-File Path Collisions**
    *   *Finding*: Checking only `.exists()` for files like `publish.yaml` or `publish.toml` can fail if a directory with the same name exists (e.g., generated by a script or editor). This would lead to unhandled directory-reading exceptions when `read_text` is called.
    *   *Suggestion*: Check `path.is_file()` instead of `path.exists()` during file resolution in both `resolve_publish_file` and `load_publish_config`.

---

## Cross-Lens Insights

*   **X1: Simpler Path Return Types Reduce Integration Risk**
    *   Changing `resolve_publish_file()` to return a complete path (instead of relative to the checked directory) improves engineering correctness (reducing coupling and relative path logic bugs in Task 3) and prevents potential silent path resolution errors for users.
