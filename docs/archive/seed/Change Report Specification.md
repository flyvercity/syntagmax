# Спецификация

## Модуль формирования отчета об изменениях между версиями документов Git

# 1. Назначение

Модуль предназначен для формирования человекочитаемого отчета об изменениях между двумя ревизиями Git.

Отчет должен анализировать изменения не только на уровне строк, но и на уровне логических объектов документа (требований, разделов и других объектов, поддерживаемых существующей утилитой анализа документов).

Результат работы представляет собой Markdown-документ, пригодный для:

* просмотра человеком;
* хранения в Git;
* публикации в Wiki;
* последующей конвертации в HTML/PDF/DOCX.

---

# 2. Цели

Функциональность должна обеспечивать:

* сравнение двух ревизий Git;
* определение измененных файлов;
* анализ изменений объектов документа;
* анализ изменений обычного текста, не принадлежащего объектам;
* формирование единого Markdown-отчета.

---

# 3. Область применения

Модуль предназначен для документов Doc-as-Code, размещенных в Git-репозитории.

Основной тип документов — Markdown.

---

# 4. Исходные данные

Пользователь задает две ревизии Git:

```
BASE
```

и

```
TARGET
```

В качестве ревизий допускаются:

* commit hash;
* tag;
* branch;
* HEAD;
* HEAD~N.

Примеры:

```
HEAD~1 HEAD
```

```
v1.2.0 v1.3.0
```

```
release develop
```

---

# 5. Используемые компоненты

Модуль должен использовать существующую утилиту объектного анализа документов.

Предполагается следующий сценарий работы.

```
Git

↓

получение измененных файлов

↓

извлечение содержимого файлов
для двух ревизий

↓

объектный анализ
(существующая утилита)

↓

сравнение объектов

↓

анализ изменений обычного текста

↓

генерация Markdown
```

Новая функциональность не должна повторно реализовывать парсер документов.

---

# 6. Входные параметры

Обязательные параметры:

```
--base <git revision>

--target <git revision>
```

Необязательные параметры:

```
--output <report.md>
```

При отсутствии параметра `--output` отчет выводится в stdout.

---

# 7. Общая структура отчета

Отчет состоит из следующих разделов.

```
# Change Report

Repository information

Summary

Changed files

Detailed changes

Appendix
```

---

# 8. Раздел Repository information

Должен содержать сведения о сравнении.

Пример:

```text
Repository

Base revision:
v1.2.0

Target revision:
v1.3.0

Generated:
2026-07-15 12:35 UTC
```

---

# 9. Раздел Summary

Должен содержать агрегированную статистику.

Обязательно выводятся (N - число):

| Параметр                | Значение |
| ----------------------- | -------- |
| Files changed           | N        |
| Files added             | N        |
| Files removed           | N        |
| Sections modified       | N        |
| Objects added           | N        |
| Objects modified        | N        |
| Objects removed         | N        |
| Text fragments modified | N        |

---

# 10. Раздел Changed files

Для каждого измененного файла выводится краткая информация.

Пример

```text
docs/system.md

Status: Modified
Objects changed:
5

Text fragments:
2
```

Статусы:

* Added
* Removed
* Modified
* Renamed

---

# 11. Детализация изменений файла

Каждый файл оформляется отдельным разделом.

```
## docs/system.md
```

Далее располагаются подразделы.

---

# 12. Изменения структуры документа

Если были изменены разделы документа, выводится список.

Пример

```text
Modified sections

2 Initialization

3 Diagnostics

Appendix A
```

---

# 13. Изменения объектов

Для каждого объекта выводится отдельный блок.

Обязательные поля:

```
Object identifier

Object type

Status
```

Пример

```text
Requirement REQ-105

Status: Modified
```

---

# 14. Изменения текста объекта

Если изменился текст объекта, выводятся обе версии.

````markdown
### Text

#### Previous

```text
The controller stores data.
```

#### Current

```text
The controller shall store encrypted data.
```
````

При наличии возможности допускается использование Markdown diff-блоков.

---

# 15. Изменения атрибутов объекта

Измененные атрибуты выводятся таблицей.

| Attribute | Previous | Current  |
| --------- | -------- | -------- |
| Priority  | Low      | High     |
| Status    | Draft    | Approved |

Неизмененные атрибуты не отображаются.

---

# 16. Изменения связей

Изменение связей отображается как изменение атрибутов объекта (смотри предыдущий раздел)

---

# 17. Изменения вне объектов

Данный раздел является обязательным.

Назначение — отображение изменений обычного текста документа, не принадлежащего логическим объектам.

Примеры:

* вводный текст;
* пояснения;
* комментарии;
* описания;
* списки;
* таблицы;
* произвольные абзацы.

---

# 18. Представление изменений обычного текста

Каждый измененный фрагмент отображается отдельно.

Обязательные поля:

* диапазон строк в исходной версии;
* диапазон строк в новой версии;
* тип изменения.

Пример

````markdown
### Text fragment

Status:
Modified

Old lines:
45-52

New lines:
45-56

#### Previous

```text
System overview.
```

#### Current

```text
System overview and operating modes.
```
````

---

# 19. Добавленный текст

Пример

````markdown
### Text fragment

Status:
Added

New lines:
132-145

```text
...
```
````

---

# 20. Удаленный текст

Пример

````markdown
### Text fragment

Status:
Removed

Old lines:
87-94

```text
...
```
````

---

# 21. Изменения файлов

Если файл полностью новый:

```
Status: Added
```

Выводится весь документ.

Если удален:

```
Status: Removed
```

Выводится содержимое предыдущей версии.

---

# 22. Обработка переименований

При обнаружении Git Rename выводится

```
Old name

docs/old.md

New name

docs/new.md
```

---

# 23. Формат Markdown

Должны использоваться:

* заголовки;
* таблицы;
* маркированные списки;
* fenced code blocks;
* горизонтальные разделители.

Не допускается использование HTML.

---

# 24. Производительность

Система должна:

* анализировать только измененные файлы;
* извлекать содержимое непосредственно из Git без переключения рабочей копии;
* использовать существующий объектный анализатор;
* выполнять сравнение объектов в памяти.

---

# 25. Обработка ошибок

Должны обрабатываться ситуации:

* отсутствует указанная ревизия;
* файл отсутствует в одной из ревизий;
* невозможно выполнить объектный анализ;
* поврежден Markdown;
* конфликт кодировки.

В случае невозможности объектного анализа файл не исключается из отчета. Для него должен быть сформирован раздел, содержащий:

* информацию об ошибке анализа;
* изменения на уровне обычного текстового diff с указанием диапазонов строк.

---

# 26. Требования к реализации

Модуль должен быть реализован как отдельная команда существующей CLI-утилиты.

Использование внешних утилит формирования diff допускается только для определения измененных файлов. Анализ содержимого должен выполняться средствами приложения.

---

# 27. Критерии приемки

Функциональность считается реализованной, если выполняются следующие условия:

1. Поддерживается сравнение любых двух ревизий Git (коммиты, теги, ветки, относительные ссылки).
2. Формируется единый Markdown-отчет.
3. В отчете присутствует сводная статистика.
4. Для каждого измененного файла отображается его статус.
5. Для каждого измененного объекта отображаются:
   * идентификатор;
   * тип;
   * статус;
   * изменения текста;
   * изменения атрибутов;
   * изменения связей (при наличии).
6. Изменения вне объектов отображаются отдельными блоками с указанием диапазонов строк в исходной и целевой версиях.
7. При невозможности объектного анализа сохраняется возможность анализа изменений на уровне обычного текста.
8. Отчет корректно генерируется для добавленных, удаленных, измененных и переименованных файлов.
9. Полученный Markdown корректно отображается стандартными средствами просмотра Markdown без использования HTML.

# 28. Краткий режим формирования отчета

Модуль должен поддерживать режим формирования краткого отчета, предназначенного для быстрого ознакомления с составом изменений без отображения их детального содержимого.

## 28.1 Назначение

Краткий отчет используется для:

- оценки масштаба изменений;
- просмотра перечня затронутых файлов;
- просмотра списка измененных объектов;
- определения необходимости анализа полного отчета.

## 28.2 Активация режима

Краткий режим включается параметром командной строки:

```text
--summary-only
```

При использовании данного параметра модуль формирует сокращенный Markdown-отчет.

## 28.3 Состав отчета

Краткий отчет должен содержать следующие разделы:

- Repository information;
- Summary;
- Changed files.

Детализированные разделы, содержащие тексты изменений, не формируются.

## 28.4 Представление изменений файлов

Для каждого измененного файла должны отображаться:

- путь к файлу;
- статус файла;
- перечень измененных объектов;
- перечень изменений обычного текста, не относящегося к объектам.

### Пример

```text
## docs/system.md

Status: Modified

Objects

+ REQ-101 (Modified)
+ REQ-102 (Added)
+ REQ-110 (Removed)

Text fragments

* Modified (lines 45-52 → 45-56)
* Added (lines 128-135)
```

## 28.5 Представление объектов

Для каждого объекта должны отображаться только:

- идентификатор;
- тип объекта;
- статус изменения.

Дополнительная информация, включая текст объекта, изменения атрибутов и связей, в кратком режиме не выводится.

### Пример

```text
Requirement REQ-205 — Modified
Requirement REQ-208 — Added
Requirement REQ-114 — Removed
```

## 28.6 Представление изменений обычного текста

Для каждого текстового фрагмента, не принадлежащего объектам, должны отображаться:

- тип изменения;
- диапазоны строк в исходной и целевой версиях.

Текст измененного фрагмента в отчет не включается.

### Пример

```text
Text fragments

Modified (lines 45-52 → 45-56)
Added (lines 128-135)
Removed (lines 210-218)
```

## 28.7 Ограничения краткого режима

В кратком режиме не должны отображаться:

- тексты объектов;
- изменения атрибутов объектов;
- изменения связей;
- содержимое измененных текстовых фрагментов;
- блоки сравнения OLD/NEW.

Краткий отчет должен содержать только информацию, достаточную для идентификации измененных файлов, объектов и текстовых фрагментов с указанием их статуса и расположения.

# 29. Формирование имен файлов отчетов

## 29.1 Общие требования

Отчет должен формироваться отдельно для каждой анализируемой секции (раздела проекта).

Для каждой секции создается самостоятельный Markdown-файл, содержащий отчет только по документам, входящим в данную секцию.

Общий отчет, объединяющий несколько секций в одном файле, не формируется.

## 29.2 Формат имени файла

Имя файла отчета должно формироваться по следующему шаблону:

```text
<section>-<date>.md
```

где:

* `<section>` — идентификатор или имя секции проекта;
* `<date>` — дата формирования отчета.

Формат даты:

```text
YYYYMMDD
```

## 29.3 Правила формирования имени секции

При формировании имени файла значение `<section>` должно удовлетворять следующим требованиям:

* соответствовать имени секции проекта;
* содержать только символы, допустимые в именах файлов;
* символы пробела должны заменяться на символ "`-`";
* недопустимые для имени файла символы должны заменяться символом "`_`" либо удаляться;
* рекомендуется использовать единый регистр (нижний или исходный), определяемый соглашениями проекта.

### Примеры

```text
system-20260715.md

software-requirements-20260715.md

flight_control-20260715.md
```

## 29.4 Каталог размещения

Все сформированные отчеты должны сохраняться в каталог, указанный параметром командной строки.

Если каталог не существует, он должен быть создан автоматически.

При отсутствии параметра выходного каталога отчеты должны сохраняться в текущем рабочем каталоге.

## 29.5 Повторное формирование

При повторной генерации отчета за одну и ту же дату файл с совпадающим именем должен быть перезаписан, если не задан иной режим работы утилиты.

## 29.6 Требования к уникальности

В рамках одного запуска утилиты имена сформированных файлов должны быть уникальными.

Уникальность обеспечивается использованием различных значений `<section>`, соответствующих секциям проекта.
