## Confirmed Bugs

### BUG-1 — `--no-git` flag is non-functional (Medium) ✅ FIXED

- **File:** `src/syntagmax/main.py` (lines 56–93)
- **Category:** logic
- **Confidence:** 98%
- **Status:** Fixed — the `process()` function now checks `config.params['no_git']` and skips the `populate_revisions` step when the flag is set.

The `--no-git` CLI flag is declared in `cli.py` and stored in the `Params` TypedDict, but was never consulted anywhere in the execution pipeline. The `populate_revisions` step (which runs `git blame` on every artifact) always executed when `impact` was in the plan, regardless of the flag.

**Runtime trigger:** `syntagmax --no-git analyze impact` — git blame no longer runs.

### BUG-2 — `lstrip('```json')` strips character set, not prefix (Medium)

- **File:** `src/syntagmax/ai_providers.py` (lines 218–220)
- **Category:** logic
- **Confidence:** 95%

The `OllamaProvider` uses `content.lstrip('```json')` to strip a markdown code fence prefix. In Python, `str.lstrip(chars)` treats the argument as a **set of characters** to strip individually, not as a substring prefix. The set `{'`', 'j', 's', 'o', 'n'}` means any leading combination of those characters is removed.

If the AI model returns JSON content that starts with characters in that set after the fence (e.g., content beginning with `"null"` or `"json_field"`), leading characters will be incorrectly stripped.

**Runtime trigger:** Ollama returns `\`\`\`json\nnull` → after lstrip, the `n` from `null` is also stripped.

**Suggested fix:** Replace with `content.removeprefix('```json')` (Python 3.9+) or use `content[7:]` after confirming the prefix with `content.startswith('```json')`.

### BUG-3 — Same `lstrip` issue in Anthropic provider (Low)

- **File:** `src/syntagmax/ai_providers.py` (lines 270–272)
- **Category:** logic
- **Confidence:** 75%

Same character-set stripping issue as BUG-2. However, the Anthropic provider has a mitigation: after the lstrip, it runs `re.search(r'\{.*\}', content, re.DOTALL)` which extracts the JSON object by matching braces. This regex fallback recovers from lstrip damage in most cases, reducing practical impact.

**Suggested fix:** Same as BUG-2 — replace `lstrip` with `removeprefix`.

### BUG-4 — Character offset used as line number in change reports (Medium)

- **File:** `src/syntagmax/change_diff.py` (lines 566–577)
- **Category:** data-integrity
- **Confidence:** 96%

The `_estimate_line_number` function uses `block.source_offset + 1` directly as a line number. However, `source_offset` is explicitly documented in `blocks.py` as a "Character offset of the opening marker tag" — not a line number. A character offset of 500 would be reported as "line 501", which is wildly incorrect.

**Runtime trigger:** `syntagmax change report --include-non-artifact --base HEAD~1 --target HEAD` on a file with marked text blocks.

**Suggested fix:** Convert the character offset to a line number by counting newlines in the source content up to that offset, or store the actual line number alongside the character offset.

### BUG-6 — Change report "Changed Files" section empty with `working` revision (Medium)

- **File:** `src/syntagmax/cli.py` (lines 540–552)
- **Category:** logic
- **Confidence:** 88%

When either `--base` or `--target` is set to `working`, the code sets `changed_files = None` and consequently `files_by_record = None`. The `ChangeReportData` then receives an empty `file_diffs` list, causing the "Changed Files" overview section to be completely absent from the report.

The artifact/text block comparisons still work (blocks are extracted from both worktree paths), but the file-level overview is missing, making the report less informative.

**Runtime trigger:** `syntagmax change report --base HEAD~1 --target working`

**Suggested fix:** For the `working` case, compute changed files using `git diff --name-only HEAD` (or the base revision) against the working directory.

### BUG-8 — SidecarExtractor passes non-string YAML values to `add_field` (Medium)

- **File:** `src/syntagmax/extractors/sidecar.py` (lines 100–107)
- **Category:** error-handling
- **Confidence:** 85%

The `add_field` method declares `value: str` in its signature, but Python doesn't enforce type annotations at runtime. The sidecar extractor passes raw YAML values (int, bool, float, None) directly from `yaml.safe_load` output. When a metamodel declares a field as `reference` with `multiple=True`, the code executes `',' in value` which raises `TypeError: argument of type 'int' is not iterable` for numeric values.

**Runtime trigger:** Create a sidecar YAML file with `priority: 3` (integer value) where the metamodel declares `priority` as a reference or multiple field.

**Suggested fix:** Wrap values with `str(value)` before passing to `builder.add_field()`.

### BUG-10 — Case-sensitive `atype` key in sidecar YAML (Low)

- **File:** `src/syntagmax/extractors/sidecar.py` (lines 98–100)
- **Category:** data-integrity
- **Confidence:** 65%

`data.pop('atype', self._record.default_atype)` is case-sensitive. If a user writes `Atype: SYS` or `ATYPE: SYS` in their sidecar YAML, the pop won't find the key, the artifact defaults to the record's `default_atype`, and the original key remains in `data` as a regular field.

This is consistent with YAML spec (keys are case-sensitive) but may confuse users.

**Suggested fix:** Either document the exact key requirement, or add case-insensitive lookup for known reserved keys.
